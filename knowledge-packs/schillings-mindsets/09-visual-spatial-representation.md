# Visual-spatial representation — code is not a chain of tokens

"Writing code or doing software engineering is **not thinking as a chain of
tokens**... I always think that code writing is a very visual activity — the
block diagram of what you're doing, or **the flow of data through your
code**. Saying that code will be just a set of tokens that you emit goes
only up to a certain point."

The innate skill being described: experts hold a *spatial* model of a
system — boxes, boundaries, flows — and derive the linear artifact (code,
prose) from it, rather than reasoning in the artifact's own serial format.
The representation is chosen to fit the problem's structure, not the output
medium.

He generalizes it to model design: Gemini was multimodal "from the onset"
because "text was only one of the modalities" — and the frontier question is
"how can a model start to think in terms of **spatial or dynamic
representations** to solve problems. That's going to become a must-have."
His slide extends it to evaluation: multimodal models allow "direct
evaluation of visual flowcharts and interface layouts against the
synthesized code" — checking the drawing against the artifact, in both
directions.

Practical form of the skill: when reasoning stalls in serial form, switch
representation — draw the data flow, then re-derive.
