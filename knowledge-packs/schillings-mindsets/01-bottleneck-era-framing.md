# Bottleneck-era framing — name the binding constraint

Schillings' signature analytical move: **periodize a field by its binding
constraint, not its technology.** He divides software into eras by what was
scarce, and reads every practice of an era as an adaptation to that scarcity:

- **Machine era** — compute was the limit; assembly-language precision and
  extracting "the last ounce of power" were rational.
- **Cloud era** — compute got cheap; the limit became **human working memory**
  ("a typical human is able to get the context between seven and nine
  tokens"). Modularity, libraries, decomposition, and code review are all
  adaptations to *that* constraint, not eternal truths.
- **AI era** — writing code stops being the constraint; pressure relocates to
  **verification and specification**: "the bottlenecks are really how do you
  ensure that that code is what you really wanted."

The transferable skill: when analyzing any practice, ask *which scarce
resource is this practice compensating for?* Then check whether that resource
is still scarce. A constraint that quietly dissolved leaves behind
load-bearing rituals that no longer earn their cost — and the next bottleneck
is already somewhere else. Model context "is basically going to be infinite
pretty soon"; processes shaped by 7±2-token humans must be re-derived, not
patched.
