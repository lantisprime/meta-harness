# Domain-tractability test — bet where verification is cheap

Why did Schillings' team bet on code in 2018, when "nobody would give us the
time of day"? Not passion — economics of the feedback loop:

- **Abundant data**: "code is a lot of data... you could go on GitHub and
  start to scrape. The amount of training data was a very unique situation."
- **Cheap verification**: "it is a domain where doing verification is
  reasonable. You can run a piece of code, you can compile it, you can have
  unit tests. The ability to figure out *is the output correct* was pretty
  reasonable to do."

"That brought us where we are today." The two-factor test — data abundance ×
verification cost — is his screen for *where progress compounds*. It also
predicts his expansion targets: chemistry and biology become tractable
exactly when "doing experiments becomes basically free," i.e. when code
makes their verification loops cheap too.

Corollary he draws explicitly: tractability is a moving property, not a
fixed one. Code's data advantage is *ending* (machine-generated code
poisons the mine), so the loop must switch from mining human artifacts to
self-play — the domain stays tractable only because verification stayed
cheap. When choosing what to learn or automate next, rank candidates by the
cost of one honest iteration of try-verify, not by how impressive the domain
sounds.
