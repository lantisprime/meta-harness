## 2. Findings (P0 → P3)

### P2

**P2-1. Synthetic PASSED validation receipts bypass `validate_policy` for the genesis; honest-but-implicent-marking.**
`src/metaharness/discovery/evolution.py:271-322` (`_build_root_activation`) mints four genuine, self-hashed `PolicyValidationReceipt(verdict=PASSED)` for SCHEMA/STATIC/SIMULATION/SHADOW **without** invoking `validate_policy` — i.e., no schema round-trip, no static lineage check, and no SIMULATION concentration/budget/candidate-node check runs (`policy.py:426-568`, especially `_validate_simulation` at `policy.py:438-567`, would legitimately FAIL the genesis because `window is None` → "a PopulationDescriptor window is required"). The genesis genuinely has no population window, so synthetic PASSED is the *only* way to receipt it — that part is justified.

Why this is only P2 and not P1: I traced the injection vectors the brief asked me to be adversarial about, and **none are realizable**:
- A non-root activation constructed in `consider()` (`evolution.py:567-580`) builds its `validation_receipt_hashes` from the receipts returned by `validate_policy(candidate, parent=parent, window=window)`, which bind `policy_hash=candidate.policy_hash` and `parent_policy_hash=parent.policy_hash` (non-None).
- `PolicyActivationReceipt._validate_stage_receipts` (`evolution.py:148-204`) requires **every** receipt's `policy_hash == self.policy_hash` (`evolution.py:166-170`) and **every** receipt's `parent_policy_hash == self.parent_policy_hash` (`evolution.py:172-176`). The bootstrap receipts carry `policy_hash=root.policy_hash` and `parent_policy_hash=None`; a non-root child has a different `policy_hash` and a non-None parent. Reuse is therefore structurally impossible — both checks fail simultaneously.
- `validation_receipt_hashes` is not consumed anywhere without its matching `validation_receipts` (the validator at `evolution.py:195-199` forces them to correspond exactly), so hash-only replay is impossible.
- The root activation cannot be replayed for a *different* root either — it binds `root.policy_hash`, and `is_root=True` requires `parent_policy_hash is None` **and** `activated_sequence == 0` (`evolution.py:178-186`), so no second/root-at-a-later-sequence forgery is constructible.
- Critically, the root was **already** active by construction before this change (`__init__` sets `self._current = root`, `evolution.py:233`); the bootstrap receipt does not *grant* new authority, it receiptises existing implicit genesis state. "Activation" here is H-plane controller state ("current policy pointer"), explicitly **not** evaluator/release/deployment authority per the `SearchPolicyEvolver` docstring (`evolution.py:210-216`).

Failure scenario (non-blocking): a **future** consumer that enumerates an evolver's validation receipts and equates "a `PolicyValidationReceipt` with `verdict=PASSED`" with "validation actually ran" would over-credit the genesis as validated. No current consumer does this (I confirmed no production code walks `activation_receipts`; see P3-3). Mitigations already present: `is_root=True`, `parent_policy_hash=None`, `activated_sequence==0`, reason strings containing "root policy bootstrap", and the SHADOW reason "grants no activation authority". Recommend: make the synthetic nature structural rather than textual — e.g. a distinct genesis `actor_label` (see P3-4) and/or a named/marked sentinel so the receipts cannot be mistaken for earned ones by a naive consumer.

**P2-2. The zero descriptor sentinel is byte-identical to the internal `_HASH_PLACEHOLDER` and is pinned by no test.**
`evolution.py:292` `zero_descriptor = "sha256:" + "0" * 64` is assigned as `descriptor_hash` on the SIMULATION (`evolution.py:293`) and SHADOW (`evolution.py:303`) bootstrap receipts. `src/metaharness/discovery/models.py:24` defines `_HASH_PLACEHOLDER = "sha256:" + "0" * 64` — the **same string**, used transiently by `_self_verifying` (`models.py:33-41`) as the placeholder it writes into a hash field before computing the real `content_hash(...)` excluding that field.

Focus-2 assessment: a real `PopulationDescriptor.descriptor_hash` is `content_hash(model_dump(exclude={"descriptor_hash"}))` (`population.py:118-128`); producing all-zeros is a SHA-256 preimage and computationally infeasible, so it cannot **collide** with a real descriptor hash. No consumer **dereferences** a `descriptor_hash` into a descriptor object — every usage either copies `descriptor.descriptor_hash` from a real descriptor (scheduler/heartbeat/policy) or is a field/hash/structural-binding (the SIMULATION==SHADOW check at `evolution.py:200-203`); I confirmed via grep that there is no `by_hash`/`lookup`/`find_descriptor` path. So the sentinel cannot **masquerade** as a validated population today.

Why P2 not P3: the shared sentinel string is fragile coupling (two independent definitions of the same magic value with no link, and `_HASH_PLACEHOLDER` is private to `models.py`), and **no test pins** the bootstrap's `descriptor_hash` value — `test_root_activation_is_the_only_is_root_receipt_and_consider_emits_non_root` (`tests/adversarial/test_discovery_policy_boundaries.py:544`) does not assert the descriptor hash. Failure scenario (non-blocking): if `_HASH_PLACEHOLDER` semantics ever change, the bootstrap's "zero" sentinel would silently diverge with no test catching it. Recommend: define one named constant (e.g. `_ZERO_DESCRIPTOR_HASH`) in `population.py`/`policy.py`, reference it from `_build_root_activation`, and add a test pinning the bootstrap receipts' `descriptor_hash` to it (plus optionally assert a real `PopulationDescriptor` never produces it).

### P3

**P3-1. The `is_root=True` validator branch has no counterexample test (would not fail if removed).**
`evolution.py:178-186` enforces `is_root=True ⇒ parent_policy_hash is None and activated_sequence == 0`. The `is_root=False` branch **is** genuinely counterexample-tested by `test_forged_non_root_activation_with_no_parent_is_rejected` (`tests/adversarial/test_discovery_policy_boundaries.py:477-542`): its first sub-case builds an `is_root=False` activation with `parent_policy_hash=None` and all-parent-None receipts and expects `ValidationError` — this sub-case would **fail** if the `is_root=False → non-None parent` invariant were removed (every other check passes: receipts' parent `None` == self parent `None`, `parent != policy`, etc.). So the brief's first question is answered YES. But a grep for `is_root=True` / `is_root = True` across `tests/**/*.py` returned **no matches**: `_build_root_activation` is the sole `is_root=True` producer and always sets the correct values, so **removing the `is_root=True` validator would not fail any test.** Recommend a counterexample test (e.g. `PolicyActivationReceipt(is_root=True, parent_policy_hash=<non-None>, …)` and `is_root=True, activated_sequence=1` → `ValidationError`).

**P3-2. No single end-to-end "inject the bootstrap receipts into a non-root activation" test (coverage is by-analogy).**
The bootstrap-injection vector is blocked, and existing tests would fail if it were accepted — `test_activation_proof_rejects_missing_failed_or_foreign_policy_receipts` (`tests/adversarial/test_discovery_policy_boundaries.py:~322-352`) rejects a foreign `policy_hash`, and the parent-`None` case above rejects the bootstrap's `parent_policy_hash=None`. So the brief's second question is answered YES (existing tests would fail). But there is **no** test that literally pulls `evolver.root_activation.validation_receipts` and attempts a non-root `PolicyActivationReceipt` for a different child, asserting rejection. Recommend a direct injection test for defense-in-depth (it would also pin the zero-descriptor + root-policy-hash binding together).

**P3-3. `root_activation` is deliberately kept out of `activation_receipts`; boundary is documented but the design rationale is test-shaped.**
`evolution.py:240-268` exposes the root via a separate `root_activation` property, with the comment that this keeps `consider()`'s appended `activation_receipts` sequence "unchanged for already-existing tests". The boundary **is** documented (`root_activation` docstring: "non-root consider() activations only"). I confirmed the specific risk the brief asks about — "any consumer, receipt chain, rollback path, or doctor/audit sweep that walks `activation_receipts` and will now silently miss the root" — is **NO**: `rollback` uses `_strategy_history` + `_known_snapshots` (`evolution.py:617-660`), never `_activation_receipts`; a grep across `src/**/*.py` shows **no production consumer** of `activation_receipts` or `root_activation` outside the evolver itself; and the root was **never** in `activation_receipts` even before `3c14884` (pre-existing), so this is not a regression. Residual concern: a **future** consumer assuming `activation_receipts` is the complete activation lineage would miss the genesis. Recommend: either prepend the root as index 0 (and update the tests, which would be the more honest audit trail) or add a `full_activation_lineage`/`all_activations` accessor plus an invariant test, so the boundary is enforced rather than merely documented.

**P3-4. `actor_label="policy-validation-gate"` on a genesis receipt that bypassed the validation gate is mildly misleading.**
`evolution.py:317-318` sets `actor_label="policy-validation-gate"` on the root activation — identical to `consider()`'s `self._ACTOR_LABEL` (`evolution.py:213`). The validation gate did **not** run for the root. An auditor correlating `actor_label` with "the gate ran" would be misled for the genesis; this slightly undercuts the `is_root`/"bootstrap" honesty marking. Recommend a distinct genesis label (e.g. `"policy-genesis-bootstrap"`) bound into `activation_hash`.
