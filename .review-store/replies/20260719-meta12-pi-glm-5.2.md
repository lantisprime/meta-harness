> P1
> 1. Evaluator may validate its own probes (charter invariant #4 violation).
>     - Evidence: `selflearn/src/selflearn/learning/improvement.py`, `assess_domain_readiness`, probe-validation block — `elif probe.validated_by == policy.optimizer_identity:` rejects the optimizer as validator but has no symmetric check rejecting `policy.evaluator_identity`. A specialist may declare `probes validated_by == policy.evaluator_identity`, pass readiness, then have that same evaluator judge its own trial eligibility via `evaluate_improvement_trial` (which only checks `trial.evaluator_identity == policy.evaluator_identity`). The evaluator thus approves the validity criteria it will be measured against.
>     - Impact: Direct breach of non-negotiable invariant "Evaluator non-self-approval"; bounded blast radius (candidate only becomes "eligible for human review," final test stays sealed), so P1 not P0.
>     - Correction: Add a branch rejecting `probe.validated_by == policy.evaluator_identity` (and treat equality with `domain_expert` consistently); cover with a test mirroring `test_readiness_rejects_probe_validated_by_optimizer`.

> P2
> 1. Frozen baseline not bound to the frozen evaluator.
>     - Evidence: `selflearn/src/selflearn/learning/improvement.py`, `_valid_baseline` — checks only pack, non-empty `model_id`, `total > 0`, `0 <= pass_rate <= 1`, and `injected is True`; never compares `data["model_id"]` to `policy.evaluator_identity`. `assess_domain_readiness` treats any injected baseline as valid regardless of which model/evaluator produced it.
>     - Impact: Readiness can be established against a baseline from a different evaluator than the one judging trials, weakening the "frozen baseline + frozen evaluator" pairing required for attribution.
>     - Correction: Require baseline `model_id == policy.evaluator_identity` (or record an explicit `evaluator_identity` field on the baseline and bind it); add a regression test.
>
> 2. `failure_clusters` is not attribution-filtered.
>     - Evidence: `selflearn/src/selflearn/learning/gaps.py`, `Learner.failure_clusters` — filters only by `failure.topic in coverage`, unlike `gap_signals` which additionally checks `retrieved_any = any(f.injected for f in fails)`. `evaluate_improvement_trial` accepts `dominant_cluster.id` as the targeting authority without verifying any failure in the cluster actually injected this specialist's pack.
>     - Impact: A "dominant verified failure cluster" used to gate eligibility can include failures where this pack's knowledge was never injected, mis-targeting improvement and crediting the candidate for an unrelated pattern.
>     - Correction: Filter `failure_clusters` to outcomes with `injected` intersecting the pack's published entries (or expose the injected/implicated evidence on `FailureCluster` so the caller can enforce it).
