### F1 — P2 (pre-existing, out of scope, but answers "is the class closed?"): unjournalled `ValueError` escape via `_ast_preflight` on a NUL-bearing source
- **Evidence:** `runtime.py:42-44` — `_ast_preflight` does `try: tree = ast.parse(source) except SyntaxError as exc: raise RuntimeCompError(...)`. The call site `runtime.py:275` (`_ast_preflight(source)`) sits **outside any try block in `run()`**, **after** the hash check at `runtime.py:266-271`.
- **Mechanism (documented CPython semantics — I did NOT execute this):** `ast.parse`/`compile` raise `ValueError: source code string cannot contain null bytes` (not `SyntaxError`) on a source string containing `\x00`. `except SyntaxError` does not catch it. `read_text(encoding="utf-8")` happily returns a `str` containing `\x00` (U+0000 is valid utf-8), and `source.encode()` round-trips it, so `actual_hash` matches `active.executor_hash` whenever the registered hash was computed over that NUL-bearing source.
- **Reachability:** narrow — requires a hash-matched NUL-bearing executor file (legitimately compiled source never contains raw NUL: the compiler embeds step data via `json.dumps`, which escapes `\x00` → `\u0000`; `compiler.py:150-166`). So this is reachable only via a tampered file+hash pair or a compiler bug, not through a legitimate compile artifact.
- **Impact:** an unjournalled raw `ValueError` escaping `run()` — an honest-termination / full-fidelity-evidence gap. It is **not** a bounded-authority breach (the path was already read under the unchanged containment check) and **not** a lineage breach (hash binding is intact). Pre-existing at `28c4b6b`; not introduced and not worsened by this diff.
- **Why it gates the verdict at WITH-MOD:** Required output #4 asks whether the unjournalled-escape class is closed for `ValueError`. Answer: **the *path-resolution* subclass is closed; the *source-NUL* (`ast.parse`) subclass is not.** Surface as a follow-up card; do not hold this diff responsible for an out-of-scope site, but do not claim the class is exhaustively closed.

### F5 — P3: predicate `ord(ch) < 32` is broader than the NUL threat but fit-for-purpose; false-refusal freedom rests on producer discipline, not a contract invariant
- **Evidence:** `runtime.py:191` `if any(ord(ch) < 32 for ch in active.path)`. Contract: `contracts.py:168-169` only `_require(bool(self.id), …)` / `_require(bool(self.pack), …)` — **no character-set constraint** on `entry_id`/`pack`; `models.py:60-61` likewise only non-empty for `ExecutorSpec.entry_id`/`.pack`. The stored `active.path` embeds them (`registry.py:206` `entry_dir = self._executors_dir / candidate.spec.entry_id`; `spec_hash` is hex-validated `models.py:64`).
- **Adversarial read (focus #3):** On POSIX, bytes `0x01–0x1f` (except `0x00`, `0x2f`) are *legal* filename components, so the predicate *could* refuse a persistable path if a `pack`/`entry_id` legitimately contained e.g. a tab. In practice no producer generates that — `distiller._entry_id` (`distiller.py:66-71`) yields `kn-<pack>-<topic>-<hex10>`, the compiler uses slug ids, and NUL (`0x00`) **cannot be a persisted POSIX path component at all** (`mkdir` would fail during `write_candidate`), so a NUL in `registry.path` is unambiguously a *tamper* signal, exactly the stated threat. DEL (`0x7f`), U+2028/2029, surrogates are **not** caught — but they also do **not** raise `ValueError` in `resolve()`/`open()`, so they don't reopen the escape. The broadened C0 rejection is reasonable hardening; just note the freedom-from-false-refusal is an empirical property of producers, not enforced by `CandidateEntry`/`ExecutorSpec`.
