FINDINGS:
- [P1] doctor.py:196-205 Orphan-source check compares incompatible path bases, false-flagging every legitimate executor. `recorded_paths` is built from `r.get("path","")`, which `registry.write_candidate` stores store-root-relative (`path.relative_to(self.store_root)` → `"<pack>/executors/<id>/<hash>.py"`, registry.py write_candidate). The disk walk computes `rel_path = str(source_file.relative_to(pack_dir))` (doctor.py:202) → `"<pack>"` prefix stripped → `"executors/<id>/<hash>.py"`. `"executors/..." in {"<pack>/executors/..."}` is always False, so every registered source is reported `executor.orphan-source` (fixable=False). On any pack that has gone through activation, `run_doctor` returns orphan findings and `DoctorReport.ok` is False, breaking the doctor's healthy-store→0 exit contract; `--fix` cannot clear them. No test exercises a registered-source pack through the doctor (the spec §5 "leaves a clean pack silent" test is absent), so CI is green but the bug is latent. Fix: `relative_to(pack_dir.parent)` or strip the pack prefix when comparing.
- [P2] gate.py:183-204 The FIX-3 "unregistered candidate" guard is dead code. `evaluate` calls `self.registry.write_candidate(candidate)` (gate.py:183) — which atomically adds the quarantined record — BEFORE searching `record_for(entry_id, status="quarantined")` for a matching `executor_hash` (gate.py:185-192). Because write_candidate always creates/finds the quarantined record, `record` is never None, so the `if record is None:` rejection at gate.py:193-205 is unreachable. This defeats FIX-3's requirement that "gate.evaluate REQUIRES an existing quarantined record … absent/mismatched → rejected, no sandbox run"; the required citing test "evaluate without prior registration → rejected, zero sandbox calls" is missing from test_compilation_gate.py. Not an authority hole (candidate is hash-bound and activation still needs sandbox+approval) but a contract-compliance defect plus missing test.
- [P2] runtime.py:49-52,71,16-18 AST preflight does not close the `getattr` sandbox escape. `_ast_preflight` rejects only `ast.Attribute` dunder access (`node.attr.startswith("__")`); it does not inspect `getattr(obj, "<dunder>")` calls. `getattr` and `hasattr` are whitelisted (runtime.py:26), and `object`/`type` are in `SAFE_BUILTINS` (runtime.py:16,18). A tampered executor (source+registry hash both compromised) can run `getattr(type(()), "__bases__")[0]` etc. to reach `object.__subclasses__()` — the very escape the preflight claims to block (F2-16) — because the access is a Call with a string, not an Attribute node. Generated compiler output is inert and hash-bound so this is defense-in-depth, but the preflight's stated guarantee is undercut and no test covers `getattr`-with-dunder-string.
- [P2] testgen.py:143 FIX-5 deviation: `approval`-kind `step_id` is not validated against the spec's step ids. `if kind == "approval": has_approval = True` (testgen.py:143-144) performs no `step_id` membership check, unlike `check` (testgen.py:145-149) and `failure-path` (testgen.py:150-153). FIX-5 explicitly required "for kinds check/approval/failure-path, step_id MUST be an actual step id in the spec." A plan entry `{"kind":"approval","step_id":"nonexistent"}` passes validation; the rendered approval test ignores step_id anyway, so it is cosmetic, but the validation gate is incomplete. (Also the `if step_id and …` guard at testgen.py:147/151 skips empty step_id, which FIX-5 also forbids.)
- [P2] testgen.py:268-278 The rendered `check`-kind test is semantically incoherent with executor check semantics, producing false sandbox rejections. The handler returns `{'status':'ok','checks':{<step_id>:<expect>}}` (testgen.py:273); the executor evaluates the step's actual `STEPS[step_id]['check']` pairs, e.g. `result.get('status')=='pass'` will fail (handler returns 'ok'), and `result.get('checks',{}).get(<check_key>)` will miss whenever the step's check key ≠ step_id. So any step with a check whose key isn't `step_id` or whose status expectation isn't `'ok'` raises StepCheckFailed → the sandbox reports fail → the gate rejects an otherwise-valid candidate. Fail-closed, but a correct executor is wrongly rejected. No test drives a `check`-kind suite against a step that has a check.
- [P2] compiler.py:75-76,86 and testgen.py:60-61,73 D6 provenance for compile/test-author is opt-in, not mandatory. `WorkflowCompiler.compile` and `WorkflowTestAuthor.author_suite` take `provenance: Optional[ProvenancePort]=None` and only journal `if provenance is not None`. D6 lists "Compile, test-author call … all appended through the injected ProvenancePort" as binding. A host that constructs these without binding provenance silently emits no compile/test-author events, violating D6. (gate/runtime correctly require provenance as positional args.)
- [P3] registry.py:33-35 unused imports `content_hash`, `canonical_json`; gate.py:11-12 unused `dataclass`, `Path`.
- [P3] runtime.py:147,260 `RunResult.status="error"` (runtime.py:260, F2-6 host-handler path) is an undocumented fourth status; spec §4 enumerated only completed/failed/awaiting_approval. Evidence-faithful (entry not implicated, outcomes=()) but the status vocabulary diverges from the spec contract.
- [P3] runtime.py:147,180-184 No-active-record refusal raises `RuntimeCompError` without journaling (D6 enumerates drift-refusal but not no-active); drift/tamper refusals journal via `self.clock().isoformat()` (runtime.py `_journal_refusal`) while runtime start/finish/approval-stop use the injected `now` string — both injected but incongruent within the same module.
- [P3] gate.py:101-110 The gate's identity re-check is a string compare on self-reported `suite.author_id == COMPILER_ID` / `not suite.identity_basis`, not a real `identity.distinct(model, compiler)` call; `IndependentTestSuite.suite_hash` (sha256 of test_source) does not cover `author_id`/`identity_basis`, so those fields are mutable without invalidating the hash. A hand-crafted `IndependentTestSuite(author_id="x", identity_basis="y")` passes the gate.
