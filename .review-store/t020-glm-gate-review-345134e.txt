**P3-1 — Taxonomy naming: dunder/`__builtins__` refusals journaled under `executor.malformed-source` (runtime.py:306-308, test_compilation_runtime.py:906).**
The new kind is honest for the NUL-byte and invalid-Python (SyntaxError) cases (the source genuinely is malformed), and it is defensible for dunder/`__builtins__` in the "non-conforming to the executor template contract" sense. However, a forbidden-dunder access (`().__class__.__bases__[0].__subclasses__()`) is semantically a *sandbox-escape attempt*, which the existing taxonomy distinguishes from mere malformation (cf. `executor.path-escape` at runtime.py for containment escapes). Journaling it as `executor.malformed-source` records an escape attempt under a "malformed" kind — the inverse of the inflation the code itself warns against at the path site ("Collapsing both into one kind would record a mere read failure as an escape attempt, corrupting the evidence taxonomy"). Severity is low because (a) the precise classification is preserved verbatim in the `reason` field (`"forbidden dunder access: __bases__"`, `"forbidden __builtins__ reference"`), so any `reason`-aware bucketing remains correct, and (b) the card explicitly licensed bringing dunder/`__builtins__` under "the same journalled contract ... under an honest `executor.*` kind" — which `executor.malformed-source` satisfies. Suggested follow-up (not required): a dedicated `executor.forbidden-access`/`executor.sandbox-escape` kind for the dunder/`__builtins__` branch to parallel `executor.path-escape`. This is a taxonomy-fidelity refinement, not a correctness or security defect; it strictly improves evidence (was: 0 journal events for these refusals → now: 1 with specific reason).

**P3-2 — Resource-exhaustion breadth: `RecursionError`/`MemoryError` from `ast.parse` on pathologically nested source would still escape unjournalled (runtime.py:294-309).**
The guard catches `ValueError` (the 3.10 NUL escape) and `RuntimeCompError` (the normalized SyntaxError/dunder/`__builtins__` cases) — exactly the targeted escape class. It does not catch `RecursionError` (a `RuntimeError` subclass, reachable from `ast.parse`'s recursive-descent/PEG parser on deeply-nested input). Such input is reachable only via a tampered hash-matched executor file, i.e. the same adversarial surface, and would breach the evidence contract for that input. This is the same shape as the explicitly-deferred P3 breadth questions (the `ord(ch) < 32` breadth, "rewriting preflight detection logic" being out of scope) and is consistent with the existing path guards, which similarly catch only `(OSError, ValueError)` at runtime.py (resolve/exists/read sites). Not a regression (pre-existing) and out of the card's scope; flagged only to keep the "is the class exhaustively closed?" answer honest. Note: `ast.walk` is implemented iteratively (deque-based), so the walker itself is not a RecursionError source; only `ast.parse` is.
