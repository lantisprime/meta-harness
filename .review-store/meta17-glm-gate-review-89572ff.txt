- **[P2]** `selflearn/src/selflearn/compilation/runtime.py:298-300` (`exec_path = Path(self.store.root) / active.path` / `source = exec_path.read_text()`) — the runtime reads executor source from `active.path` (a string read from `registry.json`) without verifying the resolved path stays inside `store_root`. Charter invariant 2 (bounded authority / no scope widening) postulates that the runtime can only reach store-local files. A tamper of `registry.json` setting `"path": "../../etc/hostile.py"` together with a matching `"executor_hash"` would widen the read surface to arbitrary files. Mitigations are real (hash check, `_ast_preflight` dunder block, SAFE_BUILTINS whitelist) so exploit impact is bounded — but the path-validation guard itself is absent. Fix: after building `exec_path`, require `exec_path.resolve().is_relative_to(self.store.root.resolve())` and refuse via `RuntimeCompError` + journal otherwise. Reasoning: the hash check stops byte-tamper, not unauthorised path inclusion. — Evidence: `runtime.py` line ~299-310 does `Path(self.store.root) / active.path` with no containment assertion; `registry.py:write_candidate` writes paths via `relative_to(self.store_root)` (safe), but reads never re-confirm containment.

- **[P2]** `selflearn/src/selflearn/compilation/gate.py:223-237` (`self.registry.write_candidate(candidate)` then `record_for(... status="quarantined")` inside `evaluate`) — FIX-3's literal text required "`gate.evaluate` REQUIRES an existing quarantined record matching the candidate's spec_hash+executor_hash; absent/mismatched → rejected receipt ('unregistered candidate'), no sandbox run." The implementation makes that rejection path unreachable by auto-registering the candidate first, so the contract clause about a 'unregistered candidate' refusal path is silently dropped. Binding invariant is functionally satisfied (a quarantined record is always present before the sandbox call), so this is a contract-clause fidelity defect rather than an authority hole. Fix: either (a) treat write_candidate as a separate orchestration step performed by the caller and require it pre-existed (raise `GateError('unregistered candidate')` instead of silently writing), or (b) document this as a deliberate simplification in the spec and remove the dead `raise GateError('candidate for {entry_id} was not registered')` branch (which today is unverifiable/unreached). — Evidence: gate.py lines ~225-237.
