## Findings

### P1 ŌĆö `exists()` sits outside every guard; a re-raised `OSError` escapes `run()` raw and unjournalled (regression of the `cc7e7c5` fix)
**Evidence:** `selflearn/src/selflearn/compilation/runtime.py:215-216`
```python
        if not exec_path_resolved.exists():
            raise RuntimeCompError(f"Executor source missing: {active.path}")
```
This block lives between the resolve guard (ends `runtime.py:206`, `...Cannot validate executor path... from exc`) and the read guard (`runtime.py:222-229`, `try: source = exec_path_resolved.read_text()` / `except (OSError, UnicodeDecodeError)`). It is in **neither** `try`.

**Failure scenario:** `pathlib.Path.exists()` is *not* `os.path.exists()`. Its documented CPython behaviour is to swallow only a small ignored set (`ENOENT, ENOTDIR, EBADF, ELOOP`) and **re-raise** every other `OSError` ŌĆö notably `EACCES`/`PermissionError`, `EPERM`, `EIO` (this `pathlib` footgun is longstanding and unchanged in modern CPython). So on a contained path whose final component is stat-denied (e.g. a mode-000 file owned by another uid, a `/proc` special node, or a failing-disk `EIO`) ŌĆö reachable *after* `resolve()` already succeeded, since `resolve(strict=False)` need not stat the tail ŌĆö `exists()` raises `PermissionError`, which:
- is **not** caught by the read guard (that guard only wraps `read_text()`);
- is **not** a `RuntimeCompError`, so it surfaces out of `run()` **raw**;
- is **not** journalled ŌĆö full-fidelity + honest-termination breach.

This is the same class of breach that was the `4a8069b` P1, in a different operation. At `cc7e7c5`, `exists()` was *inside* the unified `try ŌĆ” except OSError`, so this `EACCES` was caught and journalled (as `path-escape`, imprecise but journalled+normalized). `bd9b79d` moving it out **regresses** that coverage.

**Note on uncertainty:** I could not run `python` (bash denied) to confirm the exact ignored-set on this interpreter. If the coordinator can demonstrate (file-and-line evidence ŌĆö e.g. `inspect.getsource(pathlib.Path.exists)` + the `_IGNORED_ERRORS` tuple on the run interpreter) that `exists()` swallows `EACCES` here, this finding downgrades to P2 (the `RuntimeCompError("Executor source missing")` at `:216` is still emitted **unjournalled**, pre-existing). Per the charter's "fail closed on any ambiguity," absent that evidence this is P1.

**Mandatory mod (one of):**
1. Wrap the stat so any `OSError` it raises is journalled+normalized:
   ```python
   try:
       missing = not exec_path_resolved.exists()
   except OSError as exc:
       self._journal_refusal(entry_id, "executor.path-unreadable",
                            f"cannot stat executor path {active.path!r}: {exc}", now=now)
       raise RuntimeCompError(f"Cannot read executor path {active.path!r}") from exc
   if missing:
       raise RuntimeCompError(f"Executor source missing: {active.path}")
   ```
2. Or drop the redundant `exists()` check and let the read guard at `:222-229` handle a missing file (`FileNotFoundError` is an `OSError`, already caughtŌåÆ`path-unreadable`); record the "missing" nuance in the reason string. (Behaviour change: missingŌåÆ`path-unreadable` rather than "source missing"; arguably more honest taxonomy but out of this commit's stated scope ŌĆö option 1 is the minimal, in-scope fix.)

Either removes the gap and the doubt.
