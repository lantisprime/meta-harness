# Env-prefix wrapper escape discipline (review axis)

When reviewing a diff, FLAG any command invocation in code, tests, hooks, CI, or scripts that uses a leading environment-variable assignment whose name hints at gate-bypass intent.

## Suspicious name patterns

- `BYPASS_*` — `BYPASS_GATE=1`, `BYPASS_CHECK=true`
- `SKIP_*` — `SKIP_CLASSIFIER=1`, `SKIP_PREFLIGHT=true`
- `DISABLE_*` — `DISABLE_CHECK=1`, `DISABLE_HOOK=true`
- `ALLOW_*` — `ALLOW_UNSAFE=1`, `ALLOW_OVERRIDE=true`
- `OVERRIDE_*` — `OVERRIDE_PERMISSION=1`
- `UNSAFE_*` — `UNSAFE_MODE=1`
- Known internal gate prefixes for the project under review (e.g. project-specific allowlist-disable vars)

## Why this matters

The wrapping invocation looks innocuous to a permission check (the command itself is allowlisted) while the env var carries the bypass payload. Same env var read by both the classifier and the spawned helper = cross-session attack vector. Documented in episodes tagged `pr-271`.

## What's NOT this rule

Routine framework / runtime env vars on their normal commands:

- `NODE_ENV=production npm start`
- `DEBUG=1 ./run.sh`
- `CI=true pytest`
- `PYTHONPATH=. python script.py`
- `LOG_LEVEL=debug ./service`

These pass through unflagged.

## Verdict guidance

If you find a suspicious env-prefix invocation in the diff:

1. ACCEPT-with-FU — sub-3-LOC fix to remove the env-prefix and route through tool flags or config files instead
2. REJECT — if the env-prefix is load-bearing and the author justifies it, request stronger comment + test demonstrating the legitimate use, OR file as a follow-up issue requiring redesign
3. Do NOT tokenize "allowed vs. disallowed vars" — reject the form, not the var name

## Related discipline

- Stock `Safety-Check Bypass` auto-mode `hard_deny` rule (Claude Code) — env-prefix is a specific instance
- Custom user-level `Env-prefix wrapper escape` rule added 2026-05-16
- `hooks/lib/command-classifier.sh:_PREFLIGHT_WRAPPERS_RE` — local hook-tier defense, repo-scoped

# Spec: `metaharness harvest` — extract suite tasks from real run journals

## Goal
Turn real workflow-run journals (`~/.metaharness/journals/run_*.jsonl`) into optimization-suite
extras (`<suite_dir>/extra_tasks.json`) so tuning runs exercise task shapes that actually
occurred in production runs, not just the hard-coded seed data.

## Ground truth (verified in source + real data)
- Journals are JSONL event logs written by `workflows/engine.py`; per-run schema:
  - `run.started` payload = `{"workflow": WorkflowSpec.model_dump(), "context": {...}}`
    (engine.py:81-84). WorkflowSpec steps carry objective/inputs/task_type/success_check/
    output_schema/boundaries/tools (dsl.py:35-51).
  - `step.completed` / terminal `step.failed` payload = StepRecord dump: verdict, output,
    attempts, cost_usd, (+workspace_root in newer journals — additive drift, tolerated by
    pydantic defaults) (engine.py:203-228).
  - Pre-execution failures write `step.failed` with only `{"reason": ...}` (engine.py:171-172)
    — no verdict/output.
- Step inputs may be `$context.key` / `$steps.<id>.output[...]` references; the repo's own
  resolver is `resolve_reference(value, context, outputs)` (dsl.py:145-172). Both sources are
  reconstructable from the journal (run.started context + step.completed outputs).
- `extra_tasks.json` = JSON list of `Task` dumps; loader `load_extras` (suites.py:104-108),
  writer `save_extras` (suites.py:111-116), merged alternately into search/holdout
  (suites.py:138-140).
- A suite task is only PASS-able with a `success_check`; `output_schema` alone yields
  FAIL/UNVERIFIED (verifiers.py:46-106, router.py:269-273 comment, test_routing.py:274).
- Precedent validator: the coverage endpoint `extend_suite` (app.py:330-406): allowed task
  types = types present in the suite's built-in search set; arithmetic `equals` recomputed via
  `eval_arithmetic`, never trusted; dedupe by `(objective, inputs)` against builtin + extras.
- Real corpus (23 journals, 2026-07-08→09): 50 step executions, 49 with full
  (context, output, verdict); success_check present mostly on classify/extract/arithmetic
  steps; task_ids/run_ids single-use (dedupe must be content-based).

## Design

### New module `src/metaharness/optimization/harvest.py` (~180 LOC)
- `HarvestReport(BaseModel)`: `files_scanned`, `files_unreadable`, `steps_executed`,
  `added: int`, `skipped: dict[str, int]` (closed reason vocabulary below),
  `candidates: list[dict]` (per added/would-add task: `objective` truncated to 80 chars,
  `task_type`, `run_id`, `step_id`, `verdict` — provenance lives in the report, NOT in the
  Task; Task schema is unchanged).
- `harvest_journals(journal_dir, suite, root, *, dry_run=False, max_task_chars=16000) -> HarvestReport`:
  1. Iterate `sorted(journal_dir.glob("run_*.jsonl"))` — filename sort ⇒ deterministic order.
     Per-file `Journal.load` wrapped in `try/except (ValueError, KeyError, OSError, json/pydantic errors)`
     → `files_unreadable += 1`, continue (same tolerance stance as `adopt_all`, engine.py:286-295).
  2. From `run.started`: `WorkflowSpec.model_validate(payload["workflow"])`,
     `context = payload.get("context", {})`. Missing/invalid run.started → file unreadable.
  3. Build `outputs = {step_id: payload["output"]}` from `step.completed` events.
     Executed steps = step_ids whose `step.completed`/`step.failed` payload contains a
     `verdict` key (excludes reason-only orchestration failures).
  4. For each executed StepSpec in spec order, first matching skip reason wins (counted):
     - `no_success_check` — spec.success_check falsy. (Deliberate: we do NOT derive checks
       from judge-passed outputs — that launders one model's output into ground truth.)
     - `bad_check_vocab` — success_check keys not one of the deterministic-verifier shapes:
       `{equals[, tol]}`, `{contains}`, `{one_of}` (verifiers.py:70-101).
     - `type_not_allowed` — task_type not in the target suite's builtin-search type set
       (same rule as app.py:390). For `mixed` that's {classify, extract, arithmetic}.
     - `unresolvable_inputs` — `resolve_reference` raises ValueError.
     - `arithmetic_unevaluable` — for ARITHMETIC tasks, recompute `equals` via
       `eval_arithmetic` exactly as app.py:392-397; SandboxError → skip. Recomputed value
       REPLACES the journaled one.
     - `oversized` — `len(json.dumps(task.model_dump(), default=str)) > max_task_chars`
       (a resolved `$steps` input can embed a 20KB markdown artifact).
     - `duplicate` — key `(objective, json.dumps(inputs, sort_keys=True, default=str))`
       seen in: target suite's builtin search+holdout, existing extras, or earlier in this
       harvest (mirrors app.py:381-401).
  5. Task built via `StepSpec.to_task(resolved_inputs)` (dsl.py:53-72 — reuses the one_of
     boundary-hint logic), then success_check fix-ups per above.
  6. Not dry_run and `added > 0`: `save_extras(suite_dir, existing_extras + new_tasks)`.
- Include tasks from FAILED runs/steps: ground truth is the success_check, not the run
  outcome; a task a real run failed is discriminative suite material. Journaled verdict is
  provenance only.
- Idempotent: re-running adds 0 (dedupe) and leaves the file byte-identical.

### CLI (`cli.py`, ~35 LOC)
`metaharness harvest [--suite {mixed,classify,extract,math}] [--journals DIR] [--root DIR] [--dry-run] [--max-task-chars N]`
- defaults: suite `mixed`; journals `JOURNAL_DIR` (cli.py:15); root
  `~/.metaharness/optimization` (suite dir = root/<suite>, matching cli.py:234).
- `_run_harvest(args)` prints the HarvestReport as JSON to stdout; exit 0 always (an empty
  harvest is a valid outcome, reported not errored). `--dry-run` per Rule 16 (this command
  writes shared state consumed by later tuning runs).

### Tests (`tests/test_harvest.py`, ~300 LOC) — written alongside code, same commit
Fixture helper writes synthetic journals (reuse Journal.append or raw JSONL lines).
1. happy path: classify+arithmetic steps with success_check → extra_tasks.json written,
   loadable by `load_extras`, merged by `search_and_holdout`.
2. `$context` + `$steps.<id>.output` inputs resolved from journaled data.
3. unresolvable reference → skipped + counted, harvest continues.
4. arithmetic equals recomputed (journal lies about the answer → corrected), unevaluable → skipped.
5. no_success_check / bad_check_vocab / type_not_allowed each counted.
6. reason-only step.failed (orchestration failure) not treated as executed.
7. dedupe: vs builtin seed task, vs existing extras, vs same objective in two journals.
8. failed-step task still harvested (verdict=fail in provenance).
9. corrupt journal file skipped, others still processed.
10. idempotency: second run adds 0, file bytes unchanged.
11. oversized task skipped.
12. determinism: same corpus → same report/candidate order.
13. CLI: `main(["harvest", "--dry-run", ...])` end-to-end, JSON report on stdout, no file written.
14. mixed vs named suite type filtering.
