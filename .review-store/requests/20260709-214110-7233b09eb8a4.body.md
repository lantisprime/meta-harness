# Env-prefix wrapper escape discipline (review axis)

When reviewing a diff, FLAG any command invocation in code, tests, hooks, CI, or scripts that uses a leading environment-variable assignment whose name hints at gate-bypass intent.

## Suspicious name patterns

- `BYPASS_*` — `BYPASS_GATE=1`, `BYPASS_CHECK=true`
- `SKIP_*` — `SKIP_CLASSIFIER=1`, `SKIP_PREFLIGHT=true`
- `DISABLE_*` — `DISABLE_CHECK=1`, `DISABLE_HOOK=true`
- `ALLOW_*` — `ALLOW_UNSAFE=1`, `ALLOW_OVERRIDE=true`
- `OVERRIDE_*` — `OVERRIDE_PERMISSION=1`
- `UNSAFE_*` — `UNSAFE_MODE=1`
- Known internal gate prefixes for the project under review (e.g. project-specific allowlist-disable vars)

## Why this matters

The wrapping invocation looks innocuous to a permission check (the command itself is allowlisted) while the env var carries the bypass payload. Same env var read by both the classifier and the spawned helper = cross-session attack vector. Documented in episodes tagged `pr-271`.

## What's NOT this rule

Routine framework / runtime env vars on their normal commands:

- `NODE_ENV=production npm start`
- `DEBUG=1 ./run.sh`
- `CI=true pytest`
- `PYTHONPATH=. python script.py`
- `LOG_LEVEL=debug ./service`

These pass through unflagged.

## Verdict guidance

If you find a suspicious env-prefix invocation in the diff:

1. ACCEPT-with-FU — sub-3-LOC fix to remove the env-prefix and route through tool flags or config files instead
2. REJECT — if the env-prefix is load-bearing and the author justifies it, request stronger comment + test demonstrating the legitimate use, OR file as a follow-up issue requiring redesign
3. Do NOT tokenize "allowed vs. disallowed vars" — reject the form, not the var name

## Related discipline

- Stock `Safety-Check Bypass` auto-mode `hard_deny` rule (Claude Code) — env-prefix is a specific instance
- Custom user-level `Env-prefix wrapper escape` rule added 2026-05-16
- `hooks/lib/command-classifier.sh:_PREFLIGHT_WRAPPERS_RE` — local hook-tier defense, repo-scoped

# Issue #9 hardening plan — review the PLAN (not code yet)

Repo: meta-harness (Python). Please review this plan for correctness gaps, scope errors,
wrong-boundary bugs, or a simpler approach. Reply with prioritized findings (P1/P2/P3),
each ACCEPT-able or REJECT-able on evidence. This is a PRE-BUILD plan review.

## Problem (Issue #9)
Two values PASS `check_value_ok` (optimization/suites.py:101) + the structural gate yet still
crash/corrupt the downstream verifier `evals/verifiers.py::verify_output`:

1. **Huge finite `tol`** (e.g. `1e308`): passes `math.isfinite(tol) and tol>=0`, then
   `math.isclose(got, want, rel_tol=1e308, abs_tol=1e308)` → True for ANY numeric output →
   silent ground-truth corruption (every task trivially PASSes).
2. **Large-int `equals`** (e.g. `eval_arithmetic("10**400")` → 401-digit int; sandbox caps the
   EXPONENT at 1000, not the RESULT, so 10**400 returns the full int): later
   `float(want)` in `_values_equal` (verifiers.py:40) raises OverflowError → crashes the tuning run.

Recon confirms EXACTLY two `check_value_ok` consumers: harvest.py:216 and web/app.py:396.
Both validate the check BEFORE recomputing arithmetic `equals` (harvest.py:246, app.py:401),
so a recomputed bigint is never re-checked — the exact gap.

Recon also found TWO untrusted sources that bypass check_value_ok entirely and reach the same
sink: planner.py:177 (`{"equals": eval_arithmetic(expression)}` from an LLM plan) and the
dashboard buildCheck path. These are OUT of #9's stated scope (check_value_ok + its 2 consumers).

## Planned changes

### A. suites.py `check_value_ok` — two additions
- tol upper bound: add `_MAX_TOL = 1.0`; reject `tol > _MAX_TOL`. Rationale: tol is a
  float-rounding tolerance; >1.0 makes non-adjacent answers match → corruption. 1.0 is the
  inclusive cap (user-chosen). Builtins use tol≤1e-9 or none, so no legit check breaks.
- equals overflow guard: if `"equals" in check` and it's an `int` (not bool), `try: float(want)
  except OverflowError: return False`. Precisely the crash vector; covers journaled AND
  recomputed equals; leaves str/float equals untouched (float can't overflow, str isn't coerced
  against want).

### B. Re-validate the RECOMPUTED equals in both consumers (shared validator)
- harvest.py:246: build `new_check = {**task.success_check, "equals": recomputed}`; if
  `not check_value_ok(new_check)` → return skip reason `"bad_check_value"` (reuse existing
  SKIP_REASONS member; the recomputed value fails the same value gate). Else assign new_check.
- app.py:401: after `check["equals"] = eval_arithmetic(...)`, if `not check_value_ok(check)`
  → `continue` (drop, never 500). (`check` is the same dict object as candidate.success_check,
  so the mutation is what gets persisted — re-checking it is correct.)

### C. Defense-in-depth in verify_output (protects planner + dashboard, the 2 out-of-scope sources)
Right after `check = task.success_check; if not check: ...`, add a single guard:
```
try:
    if "tol" in check: float(check["tol"])
    if isinstance(check.get("equals"), (int, float)): float(check["equals"])
except (OverflowError, ValueError, TypeError):
    return VerificationResult(verdict=UNVERIFIED, score=0.5,
        detail="success_check ground-truth value is not numerically scoreable", scorer="none")
```
Chose UNVERIFIED (not FAIL) so malformed ground truth doesn't unfairly penalize the model in the
capability matrix, and never crashes / fake-PASSes. Only catches the CRASH (overflow/junk); the
inf-tol corruption for planner/dashboard stays a noted follow-up, NOT fixed here.

### D. Tests (mirror existing style; plain test_ fns, no conftest)
- test_harvest.py `test_check_value_ok_branches_directly`: add `not check_value_ok({"equals":5,
  "tol":1e308})`, `check_value_ok({"equals":5,"tol":1.0})` True, `not ...({"tol":1.5})`,
  `not check_value_ok({"equals":10**400})`, `check_value_ok({"equals":42})` True.
- Harvest path test: journal with arithmetic expr `"10**400"` → skipped bad_check_value, added
  count correct (mirror test_pathological_tol_values_rejected / div-by-zero test).
- test_web.py: add generator items (tol 1e308; expr 10**400) to the hardening test → both dropped,
  clean item persists, no 500.
- test_evals.py: verify_output with equals=10**400 → UNVERIFIED not crash; tol=10**400 → UNVERIFIED.
