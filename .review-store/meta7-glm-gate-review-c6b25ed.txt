### P2-1  Torn final journal write blocks all recovery (fail-closed, but unrecoverable without manual surgery)
**File:** `src/metaharness/discovery/supervisor.py:118-122` (`DiscoveryJournal.append`) and `:144-154` (`read_all`)
**Severity:** P2 — **PLAUSIBLE**
**Scenario:** `append` writes JSON + `"\n"`, `flush()`, then `os.fsync()` (`L121-122`). A **power-loss / kernel-panic between the kernel write and `fsync` completion** can leave a *partial* final line on disk (torn sector). On restart, `read_all` (`L145-150`) parses line-by-line and raises `JournalError` on the first `JSONDecodeError`. Because every line is validated (no torn-tail tolerance), a single torn final record — for an event that was **never durably committed** (fsync never acked) — poisons `recover()` for *all* subsequent restarts, even though every *prior* complete line is intact and the in-flight event's atomicity model permits it to be discarded/re-derived via the existing OMITTED/interruption synthesis.
**Charter posture:** Fail-closed (never guesses/truncates; consistent with "fail-closed validation before any… mutation" and invariant 8's immutability of prior records — they remain intact). It is an *operability* gap, not a correctness/authority violation, and does not violate any invariant as written.
**Recommendation (follow-up, non-blocking):** add torn-tail tolerance — length-prefixed records or a per-line checksum, with truncation of a single torn *final* line (safe by construction: a torn write = fsync never acked = never durably committed). Keep the strict reject for torn/corrupt *interior* lines. Especially relevant given the supervisor's raison d'être is crash recovery.

### P3-1  Knowledge evidence-receipt provenance is validate-at-append only; receipts carry `campaign_id` but no `project_id`
**File:** `src/metaharness/discovery/knowledge.py:66-82` (`_revalidate_evidence_receipt`), `append` cross-campaign check `:484-490`
**Severity:** P3 — **PLAUSIBLE** (within documented residual)
**Scenario:** `append(source_evidence_receipts=[…])` validates each receipt is a closed discovery-receipt type with a self-consistent hash and matching `campaign_id`, but does not verify the cited `receipt_id` corresponds to a real durable record, and evidence receipts carry no `project_id`, so a fabricated self-hashed receipt claiming the same `campaign_id` as a different project's campaign is not structurally excluded.
**Charter posture:** Within the explicitly-documented in-process-attacker residual ("issuance verifiers prove issuance, not cryptographic non-repudiation"; "no caller can widen scope by mere self-assertion"). The cited receipt is *not stored* (only its `receipt_id` recorded), so an auditor cross-references the durable lineage/journal log; scope-visibility itself remains issuance-gated. Acceptable for the MVP.

### P3-2  Composer identity params are caller-asserted bare strings (pending verified-assignment derivation)
**File:** `src/metaharness/discovery/contexts.py:18-32` (KNOWN LIMITATION), `compose_explorer_context` `:309`, `compose_optimizer_context` `:328`
**Severity:** P3 — **CONFIRMED** (documented `ACCEPT-WITH-MOD`)
**Scenario:** Both composers accept bare `project_id`/`campaign_id`/`lineage_id`/`parent_lineage_id`, so a caller can mis-label "I am lineage X" without independent proof. Residual is a *mis-labeled* manifest, not a *widened* one: any cross-lineage receipts produced are issuance-verified via `cross_lineage_receipt` (`:260`), and the optimizer's lineage-scoped-kind checks bind sources to the *claimed* lineage/parent. The module docstring itself recommends a dedicated follow-up (signature rework + ~30 tests). Acceptable per brief-9 disposition.

### P3-3  `make_issuance_verifier` proves issuance, not non-repudiation
**File:** `src/metaharness/discovery/supervisor.py:532-553` (`issued_assignment`, `make_issuance_verifier`)
**Severity:** P3 — **CONFIRMED** (documented known limitation)
**Scenario:** The verifier returns True iff `issued.assignment_hash == candidate.assignment_hash` against the supervisor's in-memory `_issued_assignments` (populated only once the accepted `ATTEMPT_SUBMIT_OUTCOME` is durable — `:634-636`, `:1538`). An in-process attacker with direct access to the supervisor's internals or the model constructors can forge a matching assignment.
**Charter posture:** Explicitly the documented residual threat model ("not 'no code in this process can ever lie'"). The closed-type + self-hash + same-campaign checks remain meaningful defenses. Acceptable.

### P3-4  `_commit_worktree` cannot un-create a git commit on post-commit validation failure
**File:** `src/metaharness/discovery/lineage.py:417-419` (post-commit `git status` clean-check raising)
**Severity:** P3 — **CONFIRMED** (documented known limitation, assessed acceptable)
**Scenario:** If `git status --porcelain` is non-empty immediately after `git commit` (e.g., a concurrent actor created a file between the staged-index recheck in `_stage_changes` and the commit), the manager raises *after* the commit landed, leaving a git commit with no durable `DiscoveryLineageReceipt`. In-memory `state.head_commit` is not advanced, so the next op rejects via the moved-HEAD check; `recover()` quarantines the lineage via the HEAD-mismatch anomaly detection.
**Charter posture:** Fail-closed (durable git state preserved and detected as an anomaly → quarantine; never guessed/deleted). Documented and acceptable.
