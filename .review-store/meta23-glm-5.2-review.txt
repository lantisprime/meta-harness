### P3 findings

**P3-1 — `ToolSchemaDraft` does not enforce `wire_name == name.replace(".", "__")`.**
- Evidence: `ToolSchemaDraft.validate_wire_name_matches_function` (src/metaharness/context/live.py:131-140) checks only `schema_dict.function.name == wire_name`. The registry invariant `wire_name = tool.name.replace(".", "__")` (src/metaharness/tools/registry.py:193,223) is not re-asserted on the draft. `stable_id` is built from `draft.name` (live.py:155-163) while the provider schema advertises `draft.wire_name`/`function.name`.
- Failure scenario: A hand-constructed `ToolSchemaDraft(name="a.b", wire_name="zzz", schema_dict={"function":{"name":"zzz",...}})` passes validation. The model calls `"zzz"`; `ToolRegistry.resolve_call_name("zzz")` (registry.py:189-194) returns None, so `ToolRegistry.call` raises `ToolError("unknown tool 'zzz'")` — loud, not silent. Unreachable via the production path (local.py:228-242 → `schema_records`, which always sets wire_name=name.replace(".","__")). No trust/authority/divergence/redaction/determinism impact.
- Recommended disposition: DEFER (or ACCEPT-WITH-MOD: add `wire_name == name.replace(".", "__")` to the validator to fail-fast). Does not block integration.

**P3-2 — `source_hash` semantics differ between per-tool and legacy-aggregate tool_schemas entries.**
- Evidence: per-tool path sets `source_hash_value = content_hash([raw_schema])` (src/metaharness/context/live.py:562-565), attesting the pre-redaction original; legacy-aggregate path sets `before = content_hash([dict(s) for s in tool_schemas])` then `source_hash = after = content_hash(redacted_schemas)` (live.py:590-592), attesting the redacted payload. Both set `selected_hash` to the redacted payload and `redacted = (before != after)` correctly; `ContextManifestEntry.validate_payload` (models.py:341-351) only pins `selected_hash == content_hash(payload)`, so both are model-valid.
- Failure scenario: an audit reader comparing `source_hash` to `selected_hash` to detect schema redaction gets a meaningful signal from per-tool entries (raw≠redacted when redacted) but always-equal / no signal from the legacy aggregate entry even when redaction occurred. No trust escalation, no redaction-coverage defect (the `redacted` flag and `selected_hash`/payload are correct in both), no reconstruction divergence (`reconstruct_tool_schemas` consumes `payload_json`, not `source_hash`).
- Caveat: I could not run `git diff` (sandbox blocked git), so I cannot confirm whether the legacy `source_hash=after` is pre-existing (META-19) or modified by META-23; build-spec point 7 requires the legacy raw path be "preserved," so if preserved this is a pre-META-23 consistency note rather than a META-23 regression.
- Recommended disposition: DEFER (or ACCEPT-WITH-MOD: set legacy `source_hash = before` for uniform audit semantics). Does not block integration.
